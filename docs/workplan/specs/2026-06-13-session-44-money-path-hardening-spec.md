## 1. Findings du lot (vérifiés dans le code)

### P0 — argent / quotidien

| ID | Finding | Preuve |
|---|---|---|
| **P0-A** | **Toutes les ventes sont journalisées « cash » au grand livre, quelle que soit la méthode.** Double cause, chacune suffisante : (a) `trg_create_sale_journal_entry_ins` est `AFTER INSERT ... FOR EACH ROW WHEN (NEW.status='paid')` (`20260603000014:170-174`) — il s'exécute à la fin de **l'instruction** qui pose `paid`. Or `complete_order_with_payment_v11` insère l'ordre en `'paid'` (l. 393) **avant** les `order_payments` (l. 542), et `pay_existing_order_v7` fait `UPDATE orders SET status='paid'` (l. 312) avant ses inserts payments (l. 381). Le trigger lit `order_payments` vide → la branche « split par méthode » ne produit aucune ligne → fallback systématique `SALE_PAYMENT_CASH` (le commentaire du fallback le dit lui-même : *« no order_payments rows — fallback to cash »*). (b) Même corrigé, le CASE de `create_sale_journal_entry` mappe `'debit_card'`/`'credit_card'` — valeurs **inexistantes** dans l'enum `payment_method` (`cash\|card\|qris\|edc\|transfer\|store_credit`) → `card`/`edc`/`transfer` tomberaient quand même dans le ELSE cash. La feature S26 F-AC-02 « split par méthode » est du code 100 % mort. Le report « Payment by Method » lit `order_payments` directement et paraît juste — il masque la dérive. |
| **P0-B** | **Les variantes ne sont jamais routées vers une station d'impression.** `useProducts.ts:33` exclut les enfants variantes (`.is('parent_product_id', null)`) ; `useFireToStations.ts:186-188` construit la map de routage depuis ce cache `['products']`. Une ligne issue du `VariantSelectModal` porte le `product_id` de l'**enfant** (`ProductTapHandler.tsx:76`) → non mappé → `groupItemsByStation` l'exclut de l'impression (`useFireToStations.ts:190-192`) tout en la `markPrinted`. Panier mixte : ticket imprimé **amputé** des lignes variantes (le barista prépare 2 cafés sur 3). Panier 100 % variantes prep (latte Hot/Iced…) : `firableCount === 0` → bouton « Send to Kitchen » désactivé → rien ne part en cuisine ; l'auto-fire post-paiement passe par le même trou. Le RPC, lui, route correctement (JOIN `categories` server-side, `create_variant_v1` clone `category_id`) — la perte est purement front, donc silencieuse. |
| **P0-C** | **Motif structurel : le serveur valide l'éligibilité mais fait confiance aux montants envoyés par le client.** 4 instances : (1) **montant promo** — `v11:288-345` valide active/dates/jour/heure/min/catégorie puis additionne `v_promo->>'amount'` tel quel ; une requête forgée sous JWT caissier peut effacer 100 % d'un ticket via une promo réelle au mauvais montant, **sans PIN** (le discount manuel est gaté, la promo non) ; idem `v7:189-245`. (2) **multiplier loyalty** — `p_loyalty_multiplier` calculé navigateur, utilisé brut par `v11:571` pour créditer les points (100 pts = 1 000 IDR de redemption) ; idem v7. (3) **discount de ligne appendé post-fire** — `fire_counter_order_v1` (`20260627000011:90-115`) insère `discount_amount` en ne gatant que `pos.sale.create`, jamais `sales.discount` → contournement du gate PIN S43 sur le chemin append. (4) **`change_given`** accepté verbatim par v11 (`l.375-377`) et v7 (`l.275-277`) — jamais revalidé vs `cash_received − amount` → attendus tiroir falsifiables au close-shift. |

## 2. Décisions

- **D1 (P0-A, séquencement) — Le statut `'paid'` est posé en DERNIER, après les inserts `order_payments`, dans les deux RPCs bumpés.** `complete_order_with_payment_v12` insère l'ordre en `'pending_payment'`, insère items + payments + stock, puis termine par `UPDATE orders SET status='paid', paid_at=now()` → `trg_create_sale_journal_entry_upd` se déclenche avec les payments présents. `pay_existing_order_v8` déplace son `UPDATE ... status='paid'` après sa boucle payments. Rationale vs alternative « trigger DEFERRABLE » : explicite, testable en pgTAP, pas de RAISE à commit-time qui casserait le mapping d'erreurs EF ; et on bump v12/v8 de toute façon pour D3/D4. Vérification exhaustive faite : seuls les 2 triggers JE écoutent `status='paid'` sur `orders` (le trigger expense S28 est sur `expenses`). Attention plan : tout ce qui dans v11 lit `NEW.status`/dépend de `'paid'` intra-RPC doit être re-vérifié après déplacement (gate `pending_payment` du fire, contrainte session, unicité quotidienne `order_number` — a priori neutres, l'INSERT garde les mêmes colonnes hors statut/paid_at).
- **D2 (P0-A, mapping) — Corrective de `create_sale_journal_entry` : CASE aligné sur l'enum réel + seed des mapping keys manquants.** Mapping proposé (**à valider owner/comptable au merge**) : `cash`→`SALE_PAYMENT_CASH`, `qris`→`SALE_PAYMENT_QRIS`, `card`→`SALE_PAYMENT_DEBIT`, `edc`→`SALE_PAYMENT_DEBIT`, `transfer`→clé bancaire (réutiliser le compte de `B2B_PAYMENT_BANK` S24 via une clé `SALE_PAYMENT_TRANSFER` dédiée), `store_credit`→**reste fallback cash assumé V1** (pas de ledger d'avoirs — cf. backlog). Le fallback « no order_payments rows » est conservé comme filet mais devient un cas anormal : il émet en plus un `audit_logs` `je.payment_fallback_cash` pour être observable. Vérifier en route les clés réellement seedées par S26 `_015` et ne seeder que les manquantes.
- **D3 (P0-C promo) — `evaluate_promotions_v1` devient la source de vérité du montant au paiement.** v12/v8 appellent `evaluate_promotions_v1` server-side avec le payload items de l'ordre et matchent chaque promo cliente de `p_promotions` par `promotion_id` contre le set serveur : promo cliente absente du set serveur OU `amount` ≠ amount serveur → `RAISE check_violation 'Promotion amount mismatch: <slug>'` (message distinct pour le mapping EF). Les promos serveur non soumises par le client sont ignorées (promos *dismissées* légitimes). Les gardes d'éligibilité v11 existantes restent (defense in depth). Rationale « reject » plutôt que « substituer le montant serveur » : substituer changerait le total après la saisie des tenders → mismatch tenders cryptique ; un rejet explicite est mappé proprement par l'EF et le front ré-évalue.
- **D4 (P0-C loyalty) — Le multiplier est résolu en DB ; l'argument client disparaît.** Nouveau helper SQL `get_loyalty_multiplier(p_lifetime_points INT) RETURNS NUMERIC` IMMUTABLE, miroir de `packages/domain/src/loyalty/tiers.ts` (même pattern que `get_loyalty_tier` S12, avec test de sync). v12/v8 calculent `multiplier = get_loyalty_multiplier(c.lifetime_points) × COALESCE(cc.points_multiplier, 1)` depuis `customers` + `customer_categories` et **suppriment l'argument `p_loyalty_multiplier`** de la signature (on bump, autant purger). `useCheckout` cesse de le calculer/transmettre ; l'affichage des points dans le SuccessModal bascule sur `loyalty_points_earned` de l'enveloppe serveur (déjà renvoyé, zéro consommateur aujourd'hui — fix d'affichage gratuit au passage).
- **D5 (P0-C discount append) — `fire_counter_order_v2` gate les remises de ligne.** Bump avec `p_discount_authorized_by UUID DEFAULT NULL` ajouté à la signature (+ DROP v1 même migration, REVOKE pair inline) : si une ligne du payload porte `discount_amount > 0`, l'autorisateur est requis et doit porter `sales.discount` (même gate que v11, code d'erreur `discount_requires_authorizer` réutilisé). Les call-sites (`useFireToStations`, chemin append de `useCheckout`) transmettent l'`authorized_by` déjà capturé par le DiscountModal S43 (porté par l'objet discount du cart — vérifier la forme exacte au plan). Clamp `[0, brut]` du corrective `_013` conservé.
- **D6 (P0-C change) — `change_given` revalidé server-side.** Dans v12/v8, par ligne de paiement : si `method='cash'` et `cash_received` non NULL → exiger `cash_received ≥ amount` et `change_given = cash_received − amount` ; si méthode non-cash → `change_given` NULL ou 0. Sinon `RAISE check_violation 'Invalid change amount'`. (Les gardes UI existantes restent ; elles cessent juste d'être les seules.)
- **D7 (P0-B) — Map de stations dédiée incluant les variantes.** Nouveau hook léger `useStationMap` (query key `['station-map']`) : `products.select('id, categories(dispatch_station)')` actifs **sans** filtre `parent_product_id`, staleTime aligné sur `['products']`. `useFireToStations` (firableCount au render + routage au fire via `queryClient.getQueryData(['station-map'])`) et l'auto-fire consomment cette map au lieu du cache `['products']`. Pas de changement DB. Rationale vs élargir `useProducts` : la grille DOIT continuer d'exclure les enfants variantes (design S27c) ; une map id→station séparée est le seam minimal.
- **D8 (P1-A) — Cycle de vie strict de `pickedUpOrderId`.** (a) `voidOrder()` le remet à `null` (le void serveur via `useVoidServerOrder` aboutit à `voidLocal()` → couvert) ; (b) `HeldOrdersModal` **bloque** le restore quand `pickedUpOrderId` est set (même mécanique que la garde `cartHasItems` existante, copy explicite « finish or void the current fired order first ») — on ne purge PAS dans `restoreCart` : l'ordre fired est en DB, le perdre silencieusement serait pire.
- **D9 (P1-B) — `clear()` purge le contexte client/table quand rien n'est fired.** Si `lockedItemIds` est vide après filtrage : `customerId`/`tableNumber` remis à `null` (+ `attachedCustomer`/`customerName` purgés), `order_type` resté au défaut `take_out`. S'il reste des lignes locked (ordre fired en cours), le contexte est conservé — c'est le même ordre. L'invariant K3 (lignes locked survivent à `clear()`) est inchangé ; les tests `cartStore.void.test.ts` / `networkSplit.test.ts` sont étendus, pas inversés.
- **D10 (P1-C) — Symétrie vitrine.** v8 réplique le bloc v11 `display_movements` + `display_stock` (décrément sur `is_display_item`) ; corrective `void_order_rpc_v2` (CREATE OR REPLACE, signature inchangée, pattern S38) ajoute le rétablissement `display_stock`/`display_movements` symétrique. **Tâche de vérification planifiée** : passer `cancel_item` et `refund_order_rpc_v2` au même crible (restore `current_stock` sans `display_stock` ?) et appliquer le même fix si le trou existe — scope borné aux 3 RPCs de reversal existants.
- **D11 (OPP-1) — Enveloppe de replay honnête.** Dans v12/v8, la branche replay reconstruit `change_given` (somme `order_payments.change_given`) et `loyalty_points_earned` (colonne `orders`) au lieu de NULL. `idempotent_replay: true` conservé.
- **D12 (EF) — `process-payment` bumpé et redéployé.** Appelle v12, cesse de forwarder `p_loyalty_multiplier`, mappe les nouveaux messages (`Promotion amount mismatch` → code `promo_amount_mismatch`, `Invalid change amount` → `invalid_change`) sur des codes distincts ; `classifyCheckoutError` les classe `fatal` avec copy FR (« La promotion a changé — le panier va être réévalué » → trigger une ré-évaluation au retour).

## 4. Critères d'acceptation

1. **JE par méthode** : vente QRIS → JE avec ligne au compte QRIS (pas caisse) ; vente split cash+card → 2 lignes débit aux 2 comptes ; vente cash inchangée. Le fallback « no payments » n'apparaît plus sur le chemin nominal (pgTAP) et émet un audit_log s'il se déclenche.
2. **Variantes routées** : panier 100 % variantes prep → bouton « Send to Kitchen » actif, ordre persisté, ticket contenant les lignes variantes (mock print buffer) ; panier mixte → ticket complet. Auto-fire post-paiement idem.
3. **Montants server-trusted** : `p_promotions` avec `amount` gonflé → rejet `promo_amount_mismatch` (pgTAP) ; points crédités = `FLOOR(total × multiplier_DB / 1000)` quel que soit ce que le client envoie (l'arg n'existe plus) ; `change_given` incohérent → rejet ; `fire_counter_order_v2` avec `discount_amount` sans autorisateur → `discount_requires_authorizer`.
4. **Replay honnête** : re-POST avec la même clé d'idempotence → enveloppe avec `change_given` et `loyalty_points_earned` réels (pgTAP compare 1er appel vs replay).
5. **Hygiène cartStore** : void d'un ordre fired → `pickedUpOrderId` null → le panier suivant part en v12 direct (pas de P0002) ; restore held bloqué pendant un ordre fired ; Hold puis nouveau panier → aucun client/table résiduel.
6. **Vitrine symétrique** : vente d'un `is_display_item` via v8 → `display_stock` décrémenté + `display_movements` tracé ; void → rétabli. pgTAP avant/après sur les deux chemins.
7. Sweeps verts : domain + UI + POS (baseline env-gated S25 connue exclue), `pnpm typecheck` 6/6, pgTAP nouveaux 100 %, types regen committé.
