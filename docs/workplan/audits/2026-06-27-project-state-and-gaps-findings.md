## pos-order-payment — POS order→payment (cart, orders, payments, split, held-orders)

_The POS order→payment flow is implemented robustly and is fully active: the live money path runs POS → process-payment Edge Function → complete_order_with_payment_v14 (with flag-aware stock deduction), plus a parallel fired/picked-up path via fire_counter_order_v4 + pay_existing_order_v10. Idempotency (EF UUID key + RPC p_client_uuid), split tender (max 5, overpay guards), split-by-item, manager-PIN header gating with lockout, and the held-order lifecycle (draft holds S35 + fired holds/reopen Spec A #120/#121) are all wired and look correct. The documentation, however, is badly stale: the three reference docs were last verified 2026-05-03/05-13, predate the feature-based monorepo refactor and the EF money path, and describe a non-existent architecture — wrong RPC name (complete_order_with_payments plural vs the real singular _v14), no Edge Function in the diagrams, src/hooks/pos|services/pos|components/pos paths that do not exist, a held-order model with the wrong status and hook names, and a permission taxonomy (sales.*) that mostly mismatches the code's pos.sale.* gates. The biggest risks are doc-trust (a reader following these docs would look for the wrong RPC, files, and permissions) and the completely undocumented Spec A held-order and counter fire→pay flows. No functional bug was found in code; all critical findings are documentation drift._

## production-recipes — Production & Recipes (BOM, lots, unit conversion, track/deduct flags)

_The production/recipe engine is solidly implemented and live: a single atomic RPC record_production_v1 (replaced in place by migration 20260710000024) inserts the record, raises the finished good via production_in (WAC-costed), recursively cascades material consumption to depth 5 with a cycle guard, snapshots the recipe version, optionally creates a shelf-life lot, and writes an audit row — with batch wrapping via record_batch_production_v1/v2.
