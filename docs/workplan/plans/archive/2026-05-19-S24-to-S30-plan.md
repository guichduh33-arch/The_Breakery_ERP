## 2. Découvertes alarmantes ayant motivé la priorisation S24-S30

1. **Module 09 B2B** — Surface UI shippée S14 **sans backend** : `validate_b2b_credit_limit_v1` RPC sans caller ; `B2BDashboardPage` calcule l'aging avec proxy `last_visit_at` ≠ date facture → **KPI faux en production**.
2. **Module 05 Products** — `Products.tsx` **read-only** depuis S14 : aucun RPC `create/update_product_v1`. 4 onglets `ProductDetailPage` sont des stubs. Le manager ne peut modifier le catalogue qu'en SQL.
3. **Module 10 Accounting** — Doc référence cite **11 pages BO**, le code en livre **1 seule** (`MappingsPage`). Trial Balance, GL viewer, JE saisie manuelle, AR aging absents → **audit comptable externe impossible**.
4. **Module 03 Payments** — `refund-order` EF passe le **PIN manager en clair** dans le body JSON + **aucune `p_idempotency_key`** → double-click = double refund.
5. **Module 17 Tablet** — `create_tablet_order` RPC **sans idempotency** depuis 2026-05-07 — asymétrie avec `complete_order_v9` POS.

## 8. Recommandations de gouvernance backlog

L'audit a révélé 3 sources récurrentes de drift doc-vs-code à corriger :

1. **Status notes empilées sans cleanup périodique** — Module 15 a 8 status notes périmées. **Proposition** : tous les 5 sessions, sweep automatique par agent des status notes contre les commits de livraison (pattern S30 cleanup).
2. **Doc référence Partie II "TODO à rédiger"** — Pattern observé sur `02b-orders.md`, `10-accounting-double-entry.md`, `18-mobile-shell.md`. **Proposition** : retirer les stubs aspirational dès qu'une session est jouée sans la rédiger, OU rédiger en même temps que la feature.
3. **RPCs créés sans consumer trackés** — `validate_b2b_credit_limit_v1`, `calculate_vat_payable_v1`, `get_kds_service_speed_stats`. **Proposition** : INDEX de session doit lister non seulement les migrations mais aussi les **consumers ajoutés** (apps/ files touchés), avec une section §11 "RPCs livrés sans consumer" pour les exceptions assumées.
