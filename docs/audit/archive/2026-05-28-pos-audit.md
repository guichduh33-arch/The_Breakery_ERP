## Résumé exécutif

Le POS V3 est **fonctionnellement utilisable** mais souffre de **quatre dettes critiques cachées** qui n'éclatent qu'en production : (1) le bouton *Send to Kitchen* est un placeholder client-side — le KDS ne reçoit **jamais** les commandes prises au comptoir ; (2) le drift `take_away` / `takeaway` dans 5 fichiers POS produit des branches mortes silencieuses face à l'enum DB `take_out` ; (3) les *held orders* vivent uniquement en `localStorage` et leur statut `held` n'existe pas dans l'enum DB — un crash navigateur perd la commande ; (4) le ticket imprimé hardcode `payment.method: 'cash'` et le tiroir-caisse s'ouvre quel que soit le mode de paiement. À côté, 10 findings *Major* (PIN dans body JSON pour void/cancel, absence de VirtualKeypad Qwerty, customer display sans live cart mirror, 3 onglets stub Settings, pas de scan QR/barcode, pas de vente au poids, etc.) et une dizaine de finitions *Minor* dessinent un POS qui **ressemble** à un Square mais auquel il manque encore les attaches matérielles et la rigueur DB-first qui le feraient tenir 200 transactions/jour sans flancher.

## Couche 2 — Findings priorisés

### 🔴 Critical — F-001 : `useSendToKitchen` est un no-op et le KDS ne reçoit jamais les commandes POS comptoir

- **Axe :** A, C
- **Persona :** baker cuisine + cashier en rush
- **Situation :** Cashier finalise une vente au comptoir cash takeaway → clique "Send to Kitchen" → toast "Sent 3 items to kitchen" → cliquer Checkout. Le KDS ne montre **jamais** ces 3 items.
- **Symptôme :** Le baker découvre les commandes uniquement quand le serveur passe physiquement avec un ticket papier. Pour un café/boulangerie qui veut "envoyer au boulanger AVANT de payer", la promesse `POS.md §6` est cassée.
- **Cause :**
  - `apps/pos/src/features/cart/hooks/useSendToKitchen.ts:25-39` ne fait que `useCartStore.getState().markLocked(lineIds)` — placeholder TODO admettant explicitement le no-op (lignes 8-21).
  - `supabase/migrations/20260517000015_bump_complete_order_v9.sql:375-392` insère les `order_items` **sans** `is_locked = true` ni `sent_to_kitchen_at` — defaults table = `is_locked = false`.
  - Le KDS lit `WHERE is_locked = true` (`apps/pos/src/features/kds/hooks/useKdsOrders.ts:108` — voir `.eq('is_locked', true)` ligne ~107-110). Donc *aucune* commande POS-comptoir n'est jamais affichée au KDS.
  - Seul chemin opérationnel : `create_tablet_order_v2` (S25) qui force `is_locked = true, sent_to_kitchen_at = now()` à l'INSERT (`20260602000011_bump_create_tablet_order_v2.sql:105-108`).
- **Fichiers :**
  - `apps/pos/src/features/cart/hooks/useSendToKitchen.ts`
  - `apps/pos/src/features/cart/SendToKitchenButton.tsx`
  - `supabase/migrations/20260517000015_bump_complete_order_v9.sql`
  - `apps/pos/src/features/kds/hooks/useKdsOrders.ts`
- **Invariants violés :** #3 (cart locké après envoi cuisine), #6 (commande atomique inclut envoi KDS).
- **Proposition :**
  Option A (rapide) — bump `complete_order_with_payment_v10` : à l'INSERT order_items, écrire `is_locked = true, kitchen_status = 'pending', sent_to_kitchen_at = now()` pour les items issus de la cart POS. Quasi gratuit, ferme F-001 en 1 migration.
  Option B (correcte) — implémenter `useSendToKitchen` en vrai : créer un draft order côté DB via une nouvelle RPC `create_draft_order_with_items_v1` (idempotency-keyed), garder le draft ouvert jusqu'au checkout, et faire que `complete_order` ré-utilise le draft via `pickedUpOrderId`. Beaucoup plus de travail mais débloque "Send to Kitchen ≠ Checkout" pour de vrai (`POS.md §6` mixed locked + non-locked items).
- **Effort :** Option A = S (1 migration + 1 pgTAP) ; Option B = L (1-2 sem, refacto cart + KDS + checkout).
- **Session BMAD candidate :** S31 (Option A immédiate) puis backlog (Option B).

### 🔴 Critical — F-002 : drift `take_away` / `takeaway` vs enum DB `take_out` dans 5 fichiers POS — branches mortes silencieuses

- **Axe :** D
- **Persona :** cashier + client à l'écran display
- **Situation :** Une commande créée avec `order_type = 'take_out'` (valeur réelle DB) traverse l'OrderHistoryPanel, le CustomerDisplay et l'OrderQueueTicker.
- **Symptôme :** `OrderHistoryPanel.tsx:189` affiche le mot brut `"take_out"` au lieu de "Takeaway" parce que le ternaire vérifie `'takeaway'` (qui n'arrivera jamais). `OrderQueueTicker.tsx:33` et `CurrentOrderCard.tsx:55` font la même erreur avec `'take_away'`. L'utilisateur voit du snake_case dans l'UI.
- **Cause :** Migration historique entre nomenclatures V2 (peut-être `'take_away'`) et V3 (`'take_out'`). 5 sites :
  - `apps/pos/src/features/display/components/OrderQueueTicker.tsx:33`
  - `apps/pos/src/features/display/components/CurrentOrderCard.tsx:55`
  - `apps/pos/src/features/display/__tests__/OrderQueueTicker.test.tsx:48` (fixture qui rend les tests inutiles vs réalité prod)
  - `apps/pos/src/features/order-history/OrderHistoryPanel.tsx:189` (`'takeaway'`)
  - `apps/pos/src/features/cart/HeldOrdersModal.tsx:276` (string display, cosmétique)
- **Invariants violés :** Aucun *directement* — c'est de la dette silencieuse qui camoufle un bug de données plus tard si l'enum DB est étendu.
- **Proposition :** Remplacer toutes les occurrences par `'take_out'`. Ajouter un type-narrowing helper dans `@breakery/domain/orders.ts` :
  ```ts
  export const ORDER_TYPE_LABELS: Record<OrderType, string> = {
    dine_in: 'Dine-in', take_out: 'Takeaway', delivery: 'Delivery', b2b: 'B2B',
  };
  export function orderTypeLabel(t: string): string {
    return (ORDER_TYPE_LABELS as Record<string, string>)[t] ?? t;
  }
  ```
  puis utiliser uniquement ce helper dans toutes les UI. Ajouter un test type-level qui force `ORDER_TYPE_LABELS` à couvrir tout l'enum.
- **Effort :** XS (< 1 j)
- **Session BMAD candidate :** S31 wave 1

### 🔴 Critical — F-003 : held orders en `localStorage` uniquement, statut `held` n'existe pas dans l'enum DB

- **Axe :** A, C
- **Persona :** cashier en rush + manager
- **Situation :** Le matin, un habitué commence sa commande (croissant + café), demande à attendre un ami. Cashier hold la commande pour servir le suivant. Crash navigateur ou rotation de cashier sur un autre terminal → commande perdue.
- **Symptôme :**
  - Aucune trace en DB de la commande (pas dans `orders`, pas dans `audit_logs`)
  - Un second terminal POS ne voit pas les commandes hold du premier (`useHeldOrdersStore` stockée en `localStorage` du navigateur)
  - Le manager qui ferme la shift ne peut pas réconcilier les commandes "presque finies"
- **Cause :**
  - `apps/pos/src/features/heldOrders/hooks/useHoldOrder.ts:14-17` : `useHeldOrdersStore.getState().add(toHeldOrder(cart, opts))`
  - `apps/pos/src/stores/heldOrdersStore.ts:40` : `storage: createJSONStorage(() => localStorage)`
  - Enum DB `order_status` réel : `('draft', 'paid', 'voided', 'pending_payment', 'completed', 'b2b_pending')` — pas de `'held'`
  - Documentation `POS.md §7` promet "la commande est mise en attente (statut `held`)" — promesse non tenue
- **Fichiers :**
  - `apps/pos/src/stores/heldOrdersStore.ts`
  - `apps/pos/src/features/heldOrders/hooks/useHoldOrder.ts`
  - `apps/pos/src/features/heldOrders/hooks/useRestoreHeldOrder.ts`
- **Invariants violés :** Aucun explicitement listé, mais viole le contrat métier "un comptoir ne se bloque jamais sur une commande non finalisée" qui suppose persistance et multi-terminal.
- **Proposition :**
  1. ALTER TYPE order_status ADD VALUE 'held'
  2. Nouveau RPC `hold_order_v1(p_cart_payload JSONB, p_table_number TEXT, p_notes TEXT) RETURNS UUID` — INSERT orders avec status='held' + order_items snapshot
  3. Nouveau RPC `restore_held_order_v1(p_order_id UUID) RETURNS JSONB` — désverrouille, retourne le payload pour rehydration cart
  4. Nouveau RPC `discard_held_order_v1(p_order_id UUID, p_reason TEXT) RETURNS VOID` — voided + audit_log
  5. `useHeldOrdersStore` devient un cache de `useQuery(['held-orders'])` avec realtime sub
- **Effort :** M (3-5 j)
- **Session BMAD candidate :** S31 ou S32

### 🔴 Critical — F-004 : ticket imprimé hardcode `payment.method: 'cash'` et tiroir-caisse s'ouvre toujours

- **Axe :** E
- **Persona :** cashier + manager (réconciliation cash)
- **Situation :** Client paie par QRIS → le ticket imprimé affiche "Payment: Cash" → le tiroir s'ouvre alors qu'aucun cash n'a été reçu.
- **Symptôme :**
  - **Ticket faux** — toute opération de réconciliation client/manager est compromise (un retour client basé sur le ticket "Cash" pourrait être payé en cash alors que la transaction originale était QRIS).
  - **Vulnérabilité fraude** — un cashier mal intentionné peut faire passer une vente carte mais piocher dans le tiroir qui s'ouvre, puisque le tiroir cash s'ouvre à chaque commande quelle que soit la méthode (`SuccessModal.tsx:86` : `void Promise.all([handlePrint(), openCashDrawer()])`).
- **Cause :**
  - `apps/pos/src/features/payment/SuccessModal.tsx:57-62` : champ `payment` hardcode `method: 'cash'`, `cash_received: props.cashReceived`, `change_given`. Aucun input pour les autres méthodes.
  - `apps/pos/src/features/payment/SuccessModal.tsx:86` : `openCashDrawer()` est dans un `Promise.all([handlePrint(), openCashDrawer()])` inconditionnel.
  - `apps/pos/src/services/print/printService.ts:26` : type `payment.method` est `'cash'` literal — n'accepte rien d'autre.
- **Fichiers :**
  - `apps/pos/src/features/payment/SuccessModal.tsx`
  - `apps/pos/src/services/print/printService.ts`
- **Invariants violés :** Aucun nommé, mais viole le contrat sécurité "anti-fraude" implicite des invariants 3 et 6.
- **Proposition :**
  1. `ReceiptPayload.payment` doit accepter une liste de tenders : `payment: Tender[]` (méthode + montant + ref). Adapter `buildReceiptPayload` pour lire `props.tenders`.
  2. `SuccessModal` : `useEffect(() => { if (cashTenderPresent) void openCashDrawer(); void handlePrint(); }, [...])` — drawer conditionné.
  3. Receipt template côté print server doit rendre toutes les méthodes (Cash 50k, QRIS 100k, Card 50k).
  4. Tests : ajouter cas split cash+card dans `print.smoke.test.tsx`, vérifier que `openCashDrawer` n'est pas appelé pour QRIS-only.
- **Effort :** S (1-2 j)
- **Session BMAD candidate :** S31

### 🟠 Major — F-005 : `VirtualKeypadProvider` absent — invariant n°2 « touch-first » violé en pratique

- **Axe :** A, C
- **Persona :** cashier
- **Situation :** Lors d'une recherche client par nom dans `CustomerAttachModal`, ou pour taper des notes spéciales sur un item, il faut un clavier physique.
- **Symptôme :** Le matériel sur le comptoir doit inclure un clavier (hygiène, ergonomie, encombrement). Quand le terminal est un iPad, le clavier iOS natif apparait — mais il occupe 50% de l'écran et masque les résultats de recherche.
- **Cause :** Le doc `POS.md §11` annonce `<VirtualKeypadProvider>` enveloppant `/pos`, avec layouts `NumpadLayout` + `QwertyLayout`. Aucun composant `VirtualKeypadProvider` ou `QwertyLayout` n'existe en V3. Seuls existent : `Numpad`, `NumpadPin`, `NumpadVirtual` (modaux ponctuels pour montant et PIN).
- **Fichiers manquants :**
  - `packages/ui/src/components/VirtualKeypadProvider.tsx` — à créer
  - `packages/ui/src/components/QwertyLayout.tsx` — à créer
  - Hook `useVirtualKeypad` à câbler dans `CustomerAttachModal`, `DiscountModal` (reason), `CancelItemModal` (reason)
- **Proposition :** Construire `VirtualKeypadProvider` qui intercepte le focus sur les `<input>` à l'intérieur de `/pos`, affiche un clavier overlay (numpad ou qwerty selon `inputmode`/`type`). Optionnel : permettre opt-out via `data-no-vkp`.
- **Effort :** M (3-5 j)
- **Session BMAD candidate :** S32

### 🟠 Major — F-006 : `useVoidOrder` et `useCancelOrderItem` envoient `manager_pin` dans le body JSON (viole convention S25 PIN-en-header)

## Annexe A — Innovations (5)

### Innovation 1 — Hold orders DB-backed avec snapshot atomique
Au-delà de fermer F-003, l'introduction d'un status `held` avec snapshot JSONB (sur le modèle de `recipe_versions.snapshot` S15) débloque la **vraie promesse** "comptoir ne se bloque jamais" : un manager peut sur n'importe quel terminal voir l'ensemble des commandes hold actives, les transférer entre terminaux, ou les solder à la fin de service.

### Innovation 2 — Quick-Add "Café + ce produit" — upsell bakery-native
Sur chaque CartLineRow d'un produit boulangerie (catégorie pain / viennoiserie), proposer un petit chip "+ Café" qui ajoute le café signature pour 25k. Désactivable per-product via une nouvelle col `products.upsell_partner_id`. Augmente le panier moyen sans formation cashier.

### Innovation 3 — Live cart mirror Customer Display (F-007 inversé en feature)
Plutôt que juste afficher le cart, le Customer Display devient un terrain de confirmation : flash + son à chaque ajout, allergen badge (revisitable `project_allergens_wontfix`), bouton "Add a coffee?" tappable depuis l'écran client. Boost adoption display + résout F-007.

### Innovation 4 — Auto-detection iPhone NFC pour fidélité
Beaucoup d'habitués utilisent leur iPhone. Activer une lecture NFC passive sur le pad de paiement (Apple Wallet pass The Breakery) qui auto-attache le client à la commande sans clic. Compense l'absence d'Apple Pay tap-to-pay (G4) tout en améliorant la liaison loyalty.

### Innovation 5 — Tablet handover seamless dine-in
Sur dine-in : le serveur prend la commande à table avec sa tablette (existe). Au checkout, le serveur génère un QR sur sa tablette → le client scanne avec son téléphone et paie via QRIS. Aucun cashier requis, le serveur garde le contrôle. Couvre F-009 / G9 d'un coup.
